## 5. 云图库高并发改造路线

目标：把云图库从普通业务项目升级成能讲高并发、缓存、异步、限流和压测的后端项目。

### 5.1 热门图片多级缓存

设计：

```text
Caffeine 本地缓存 -> Redis 分布式缓存 -> MySQL
```

重点补齐：

- 缓存穿透：布隆过滤器 / 空值缓存
- 缓存击穿：互斥锁 / 逻辑过期
- 缓存雪崩：随机过期时间
- 缓存一致性：更新 DB 后删除缓存 / 延迟双删
- 热点 Key：本地缓存兜底

简历口径：

```text
针对热门图片详情和空间信息高频访问场景，引入 Caffeine + Redis 多级缓存，设计空值缓存、随机过期和热点 Key 本地兜底策略，并通过压测对比缓存命中率与接口耗时。
```

### 5.2 图片上传异步处理

适合拆异步的操作：

- 图片压缩
- 缩略图生成
- 图片审核
- AI 扩图
- 标签解析
- 元信息补全

推荐链路：

```text
用户上传 -> 写入图片基础信息 -> 投递任务 -> Worker 异步处理 -> 更新任务状态
```

重点准备：

- 为什么异步
- 重复消费怎么办
- 任务失败怎么重试
- 状态表怎么设计
- 用户如何查询处理进度

如果暂时不接 Kafka/RabbitMQ，可先用：

- Redis Stream
- DelayQueue
- 线程池 + 任务表

面试口径：

> 项目中先用任务表和线程池模拟异步队列，方案可以平滑替换为 MQ。

### 5.3 接口限流与防刷

重点接口：

- 上传接口
- 搜索接口
- AI 扩图接口
- 高频详情接口

实现方案：

- Redis + Lua 滑动窗口
- Guava RateLimiter
- Sentinel

需要讲清楚：

- 固定窗口 vs 滑动窗口
- 单机限流 vs 分布式限流
- 用户维度 / IP 维度 / API 维度
- 超限返回策略

### 5.4 搜索接口优化

云图库天然适合讲搜索和查询优化。

重点场景：

- 关键词搜索
- 标签搜索
- 颜色相似度
- 以图搜图
- 空间维度过滤
- 分页和排序

需要准备：

- 索引怎么建
- 为什么不能深分页
- 怎么做游标分页
- 模糊查询如何优化
- 热门检索结果怎么缓存

### 5.5 权限校验缓存

云图库的公共空间、私有空间、团队空间是很好的后端设计点。

重点模型：

```text
用户 - 空间 - 角色 - 权限
```

优化方向：

- RBAC 权限模型
- 方法级鉴权
- AOP 注解鉴权
- 权限结果缓存
- 角色变更后的缓存删除

必须能回答：

- RBAC 和 ACL 区别
- 团队空间成员权限怎么隔离
- 权限缓存会不会导致越权
- 成员角色变更后缓存怎么失效

### 5.6 分表与索引优化

简历中写了 ShardingSphere，就必须能讲透。

建议口径：

> 图片数据天然按空间归属隔离，因此优先以 spaceId 作为分片键，使团队空间内图片查询尽量路由到单表；对于全局广场类查询，保留冗余索引表或搜索引擎承接。

需要准备：

- 为什么分表
- 分片键为什么选 spaceId
- 跨表查询怎么处理
- 分页怎么做
- 后续扩容怎么做

### 5.7 幂等与重试机制

重点场景：

- 重复上传
- 重复消费
- 重复回调
- 任务失败重试

方案：

- 业务唯一键
- 幂等 token
- 唯一索引
- 任务状态机
- 状态流转 CAS

### 5.8 压测与性能报告

这是弥补“实习没有高并发经历”的关键。

压测工具：

- JMeter
- wrk
- ApacheBench
- Gatling

压测接口：

- 图片详情

- 上传接口
- 空间图片列表
- 权限校验接口

记录指标：

- QPS
- 平均耗时
- P95 / P99
- 错误率
- 缓存命中率
- CPU / 内存
- 数据库慢 SQL

压测报告结构：

```text

并发数

优化后

```

## 8. 欧莫威实习后续争取方向

需要主动争取这些任务：

- 任务状态机：Pending / Running / Success / Failed / Retrying
- Checkpoint：记录每个阶段输入、输出、失败原因
- 产物管理：每次生成的 txt / md / xlsx / arxml / xdm 有版本
- 失败重试：支持从失败节点继续
- 日志追踪：每次 LLM 调用、解析、合并都有 traceId
- 简单 API：本地 HTTP 服务或可被流水线调用的服务接口

可以和 Leader 这样表达：

> 我想把当前 LIBAI / PTS-AI 的流程抽象成任务化执行模型，支持任务创建、进度查询、失败恢复、结果追踪和产物管理。这样后续接 Web 页面、自动化流水线或批量处理时会更稳定。

如果能做出来，简历可以从“工具开发”升级成：

> AI 工具链任务执行平台

你的路线应该是：

```text
AI 应用后端 + 高并发基础扎实 + 有复杂任务流经验的后端候选人
```

- 欧莫威：任务流、可靠性、AI 工程化
