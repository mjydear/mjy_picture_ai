## 1. 全局约定

- **主线场景**：以「热门公共图库浏览 + 图片上传」作为高并发主场景，所有优化围绕它展开并用压测量化。
- **每步闭环**：`实现 → 自测 → 压测/验证 → 记录数据 → 本地提交`（需要推送远程时单独确认）。
- **提交规范**：沿用 Conventional Commits，例：`feat(cache): add multi-level cache with logical expiration`。
- **可复现**：所有中间件用 `deploy/docker-compose.yml` 一键启动，配置入 `application-perf.yml`，用 `perf` profile 隔离。
- **不堆技术**：每个中间件都要能回答“为什么用、不用会怎样、有什么坑、压测数据多少”。

## 4. 阶段 P0：压测基线与可观测地基

> 目的：先有“优化前”的数据，否则一切优化都无法量化。**这是整套计划里性价比第一的前置工作。**

### Step 0.1　造测试数据 ✅
- **目标**：把 `picture` 表灌到 50w~100w 量级，模拟真实数据规模。
- **落地点**：`mjy-picture-backend-ddd` 下新增 `sql/perf_seed.sql`（存储过程批量插入）或一个一次性 `PerfDataSeeder` 测试类。
- **关键技术**：MySQL 存储过程 / `rewriteBatchedStatements=true` 批量插入。
- **操作步骤**：
  1. 准备随机标题、分类、标签、url、宽高、大小等字段生成逻辑。
  2. 分批插入（每批 1000~5000），避免大事务。
  3. 校验数据量与索引。
- **验收标准**：`select count(*) from picture` ≥ 50w；分页查询能复现慢的现象。
- **完成情况（实测）**：脚本 `sql/perf_seed.sql`（MySQL 存储过程，单批 2000、共 250 个事务）已执行，灌入 **50 万条**公共图库数据（`spaceId=NULL, reviewStatus=1`），耗时 **35.7s**；9 大分类均匀分布（各约 5.5w）。执行命令 `mysql -uroot -p**** yu_picture -e "source sql/perf_seed.sql"`。

### Step 0.2　压测基线 ✅
- **目标**：对核心接口压测，产出「优化前」基线报告。
- **落地点**：新增 `perf/` 目录，放压测脚本与报告。
- **工具选择（Windows 友好）**：优先 **k6**（脚本化、报告清晰）或 **JMeter**（已装 Java 即可）；`wrk` 需 WSL，可选。
- **压测接口**：图片分页列表、图片详情、登录、（上传放 P2 前后对比）。
- **指标**：QPS/TPS、P95/P99、错误率、CPU、GC。
- **操作步骤**：
  1. 写 k6 脚本（分页、详情各一个场景，含参数随机化）。
  2. 固定并发梯度（如 50 / 200 / 500 VU）。
  3. 跑基线，导出报告到 `perf/baseline/`。
- **验收标准**：拿到每个接口的基线 QPS 与 P99，并填入第 9 节对比表「优化前」列。
- **完成情况（实测，50 VU × 30s，k6 v2.0.0）**：
  - 脚本：`perf/baseline/list_vo.js`（列表无缓存）、`detail_vo.js`（详情）、`list_vo_cache.js`（项目现有缓存版对照）。
  - 列表 `POST /picture/list/page/vo`：**QPS≈2，P95=28.7s，P99=45.0s**，业务错误率 0%（50VU/30s 仅完成 97 请求）。
  - 列表缓存版 `.../vo/cache`：**QPS≈1.8，P99≈39.7s** —— 现有缓存对「随机分页」key 过于分散，命中率≈0，几乎无提升（重要发现）。
  - 详情 `GET /picture/get/vo`：HTTP 200 但 `code≠0` 占 100% —— 接口需登录态，匿名被 Sa-Token 拦截，**详情真实基线待带 token 重测（顺延到 P1.1 详情缓存）**。

### Step 0.3　瓶颈定位 ✅
- **目标**：找出基线慢在哪（SQL？序列化？无缓存？）。
- **关键技术**：MySQL `profiling` / `EXPLAIN`、k6 单并发对照、复合索引验证。
- **完成情况（量化根因，已按真实请求修正）**：

| 诊断 | 数据 | 结论 |
| --- | --- | --- |
| k6 50 VU 列表 | **QPS≈2，P99=45.0s** | 接口慢由分页 `COUNT(*)`、排序场景 filesort、SQL 明细日志竞争共同放大 |
| 真实请求无排序 SQL | `LIMIT 12000,12` 单独 **0.23s** | 基线脚本未传 `sortField`，无排序深分页单独不是头号瓶颈 |
| `COUNT(*)` | 冷态 **8.3s**；`ANALYZE` 后仍 **2.45s** | 每次分页都 count 50w，当前无排序接口的头号瓶颈 |
| `ORDER BY createTime DESC LIMIT 0,12` | **6.5s** + filesort | 前端/默认排序场景会触发 50w 级排序 |
| 加 `(spaceId, reviewStatus, createTime)` 后排序首页 | **0.0103s（约 635×）**，`Backward index scan` | 复合索引消除 filesort，排序 SQL 基本解决 |

**瓶颈清单（指导 P1，按收益排序）**：
1. **分页 `COUNT(*)` 全扫 / 慢扫**（冷态 8.3s，预热后仍 2.45s）→ 缓存 total / 热门列表整页缓存 / 必要时改为首屏精确、深页估算。【P1.1】
2. **排序场景 filesort**（6.5s→0.0103s）→ 建 `(spaceId, reviewStatus, createTime)` 复合索引，消除排序。【P1.0】
3. **无有效缓存**：每次都打 DB；现有 cache 版 key 过散命中率≈0 → 多级缓存 + 合理 key 粒度（热门页/详情）。【P1.1/1.2】
4. **并发竞争放大**：`sql-show=true` + `StdOutImpl` 同步日志抢 `System.out` 锁 → 关 SQL 明细日志。【P1.0】
5. **深分页体验风险**：无排序 SQL 单独不慢，但排序/筛选组合下仍可能退化 → 键集分页（`where createTime < ?`）/ 游标。【P1.3 榜单 + 列表改造】
- **验收标准**：输出一份「瓶颈清单」，指导 P1 优化方向。✅

## 5. 阶段 P1：多级缓存 + 缓存三大问题 + 限流 + 监控（最高性价比）

### Step 1.0　索引与 SQL 日志优化 ✅
- **目标**：先做最低风险、最高确定性的 DB 与配置优化，为后续缓存压测建立干净环境。
- **落地点**：`sql/perf_index.sql`、`src/main/resources/application.yml`。
- **关键变更**：
  1. 新增索引 `idx_space_review_ct(spaceId, reviewStatus, createTime)`，覆盖公共图库/空间图库的等值过滤 + 时间排序。
  2. `shardingsphere.props.sql-show=false`，关闭 ShardingSphere SQL 打印。
  3. MyBatis `log-impl` 改为 `NoLoggingImpl`，关闭 `StdOutImpl` 明细日志，避免高并发下抢 `System.out` 锁。
- **验证结果（实测）**：
  - SQL 排序场景：`ORDER BY createTime DESC LIMIT 0,12` **6.5s → 0.0103s**，EXPLAIN 显示 `Backward index scan`，无 `Using filesort`。
  - 后端重启后 `/api/doc.html` 返回 200；服务日志不再输出 `==> Preparing` / `<== Row` 明细。
  - k6 50 VU × 30s：无排序列表 **QPS≈1.53，P99=31.85s**；排序列表 **QPS≈1.37，P99=36.36s**。接口仍慢，原因是分页 total 的 `COUNT(*)` 尚未缓存。
- **结论**：P1.0 已解决排序 SQL 与日志竞争；真正的接口级质变需要 P1.1 缓存列表页与 total。

### Step 1.1　多级缓存组件 ⬜
- **目标**：把现有 Redis + Caffeine 雏形升级为统一的 L1(本地)+L2(分布式) 缓存能力。
- **落地点**：`shared` 层新增 `cache` 包（如 `MultiLevelCache`、`CacheService`）。
- **关键技术**：Caffeine（L1，容量+TTL）、Redis（L2），读穿/回填、统一 Key 规范。
- **应用接口**：图片详情、空间信息、热门列表。
- **验收标准**：详情接口缓存命中率 ≥ 95%，P99 显著下降并记录。

### Step 1.2　缓存三大问题防护 ⬜
- **目标**：解决穿透 / 击穿 / 雪崩。
- **关键技术**：
  - 穿透：布隆过滤器 + 空值缓存（短 TTL）。
  - 击穿：热点 Key **逻辑过期** + 互斥重建（或 Redisson 互斥锁）。
  - 雪崩：TTL 加随机抖动 + L1 兜底。
- **落地点**：并入 1.1 的缓存组件。
- **验收标准**：构造不存在 Key 高并发压测不打到 DB；热点 Key 重建期间无抖动。

### Step 1.3　热门图库排行榜 ⬜
- **目标**：高并发读场景（榜单纯走缓存）。
- **关键技术**：Redis ZSet 维护热度；浏览量 Redis 计数 + 定时落库。
- **落地点**：`domain/picture` 增加热度维度；定时任务刷新。
- **验收标准**：排行榜接口万级 QPS、零错误。

### Step 1.4　限流 / 降级 / 熔断 ⬜
- **目标**：突发流量平滑拒绝，核心接口不被拖垮。
- **关键技术**：Sentinel（带控制台）；分布式限流可用 Redis + Lua 令牌桶；COS / AI 扩图调用加熔断降级。
- **落地点**：上传、登录、搜索接口配置规则；第三方调用包降级兜底。
- **验收标准**：超阈值返回限流提示而非雪崩；Sentinel 控制台可演示。

### Step 1.5　监控大盘 ⬜
- **目标**：实时可观测，面试可截图。
- **关键技术**：Micrometer + Prometheus + Grafana；自定义指标（QPS、延迟、缓存命中率、限流次数）。
- **落地点**：引入 `micrometer-registry-prometheus`，暴露 `/actuator/prometheus`；导入 Grafana 面板。
- **验收标准**：Grafana 看到 JVM + 接口 + 缓存命中率实时曲线。

## 6. 阶段 P2：上传异步化 + MQ + 分布式锁 + 幂等

> 执行到本阶段时把每步细化到操作级（同 P0/P1 模板）。

### Step 2.1　上传链路异步化 ⬜
- 上传拆分：快速接收 → 异步处理（缩略图、主色调、入库）；自定义线程池 + `CompletableFuture`。

### Step 2.2　MQ 削峰 ⬜
- 引入 RocketMQ：上传事件入队，消费者异步落库与处理；高峰削峰保护 DB。

### Step 2.3　分布式锁 ⬜
- Redisson：协同编辑跨节点抢占（升级现有 `ConcurrentHashMap` 方案）、空间容量/数量扣减防超卖。

### Step 2.4　幂等与去重 ⬜
- 文件 MD5 秒传去重；接口幂等 token 防重复提交。

## 10. 面试故事卡（边做边攒）

1. **缓存击穿**：热门图片详情高并发 → 逻辑过期 + 互斥重建 → P99 80ms→5ms，命中率 96%。
2. **上传削峰**：高峰上传打满 DB → MQ 异步化 + 线程池 → 接口 600ms→60ms，DB 平稳。
3. **限流降级**：突发流量 → Sentinel QPS + 热点限流 + COS 熔断降级 → 核心接口可用性 100%。
4. **分库分表**：百万级图片 → 按 spaceId 分表 → 单表数据量可控，分页 P99 稳定。
